### ISSUE-1: Worker-daemon authentication model is still contradictory
- Category: architecture
- Severity: medium
- Problem: Section 7 says internal worker-daemon endpoints authenticate with `X-Worker-Token` ([spec](/Users/sunsilkk/service/ml_management_v2/docs/specs/2026-04-10-ml-training-platform-design.md#L958)), but Section 13 still says the worker daemon authenticates to the API with a long-lived service-account JWT ([spec](/Users/sunsilkk/service/ml_management_v2/docs/specs/2026-04-10-ml-training-platform-design.md#L1365)).
- Why it matters: The runtime/security contract still does not define a single credential model, so implementation, rotation, storage, and revocation semantics remain ambiguous.
- Suggested fix: Either pick one mechanism for all worker-daemon API calls, or explicitly split the model by surface (for example `X-Worker-Token` for `/api/v1/internal/*`, JWT for other control-plane endpoints) and update Section 13 to match.

### VERDICT
- Status: REVISE
- Reason: The plan is nearly consistent, but the worker-daemon auth model still conflicts between the API and security sections.
