## Issues Disputed

### ISSUE-1: Orchestration model is internally inconsistent [DISPUTED]
The three sections describe different abstraction layers, NOT contradictions:
- Section 3 (Worker System): Describes the DAEMON-LEVEL lifecycle — daemon receives Celery task, spawns container, manages per-job Docker lifecycle. "Pull from MinIO" and "Upload results" here refer to daemon actions (daemon has network), NOT container actions.
- Section 4 (SALAD Pipeline): Describes INSIDE the container — salad_pipeline library + FilesystemCallback writing to /workspace/ipc/.
- Section 5 (IPC Architecture): Explicitly bridges both layers — container writes IPC files, daemon watches and forwards to MinIO/Redis/API.
The plan DOES state one execution model: Celery dispatches task → daemon spawns air-gapped container → container writes IPC → daemon forwards. This is consistent across all three sections.

**However** — I accept that the clarification could be made more explicit. I've added an "Execution ownership" block at the top of Section 3 that explicitly names Celery as dispatch-only, daemon as execution owner, and DB as state source of truth.

### ISSUE-7: Worker admission ignores local staging capacity [DISPUTED — LOWER SEVERITY]
Section 13 already specifies `--memory`, `--cpus`, `--shm-size` set per worker config. The daemon allocates these at container creation time. Disk capacity is bounded by worker config + tmpfs size limits. For Phase 1 with 2-3 workers, these are admin-configured values. Full dynamic admission with preflight estimates is over-engineering for this scope. I downgrade this to LOW severity (operational concern, not architectural gap).

### ISSUE-8: Control-plane authorization underspecified [PARTIALLY ACCEPTED]
The plan DID already describe X-Worker-Token in Section 13 (line 1369): "Worker daemon authenticates to API with X-Worker-Token header (pre-shared secret per worker, stored hashed in workers table)". However, I accept that mutation-scope binding was missing. I've added: (1) `token_hash` column to workers schema, (2) Authorization binding rules (X-Worker-Token + X-Attempt-Id + lease ownership check) to internal API section.
