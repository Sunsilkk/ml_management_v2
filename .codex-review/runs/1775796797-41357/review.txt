### ISSUE-1: Worker-daemon write-back endpoints are still undefined
- Category: architecture
- Severity: medium
- Problem: The IPC watcher flow now says it updates `job_steps` via API, forwards eval results via API, and calls `PATCH /api/v1/training/jobs/{job_id}/steps/{step_id}` after checkpoint upload ([spec](/Users/sunsilkk/service/ml_management_v2/docs/specs/2026-04-10-ml-training-platform-design.md#L601), [spec](/Users/sunsilkk/service/ml_management_v2/docs/specs/2026-04-10-ml-training-platform-design.md#L611)), but Section 7 only documents read-only step/metrics/evaluation endpoints plus job-level create/cancel/resume ([spec](/Users/sunsilkk/service/ml_management_v2/docs/specs/2026-04-10-ml-training-platform-design.md#L941)).
- Why it matters: The new IPC bridge cannot be implemented against the documented platform contract because the worker daemon has no defined write path for step events, checkpoint metadata, or evaluation inserts.
- Suggested fix: Add explicit internal worker-daemon endpoints to Section 7, or state that the watcher writes through a non-HTTP service/DB boundary instead of the public API, and define the auth model for that path.

### VERDICT
- Status: REVISE
- Reason: The plan is close, but the IPC runtime still depends on write-back interfaces that the API section does not define.
